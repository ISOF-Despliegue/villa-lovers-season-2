# Resumen del proyecto StreamButed

**StreamButed** será una plataforma distribuida de streaming de audio, pensada como una alternativa académica similar a Spotify o SoundCloud, pero construida con una arquitectura moderna orientada a resolver problemas comunes en sistemas monolíticos, como la alta latencia, los cuellos de botella en la red y los puntos únicos de fallo. En lugar de concentrar toda la lógica en una sola aplicación, el sistema se dividirá en varios servicios independientes, cada uno especializado en una función concreta. 

La idea principal del proyecto es permitir que distintos tipos de usuarios interactúen con la plataforma. El **oyente** podrá buscar canciones y reproducirlas por streaming; el **artista o creador** podrá subir archivos de audio e imágenes junto con sus metadatos; y el **administrador** tendrá funciones para moderar contenido, consultar analíticas y gestionar usuarios o publicaciones. De esta forma, el sistema cubrirá tanto el consumo de contenido como su administración y publicación.

El proyecto se hará siguiendo una arquitectura de **microservicios basada en Domain-Driven Design (DDD)**. Eso significa que el sistema se dividirá por dominios de negocio bien definidos. Por ejemplo, habrá un servicio exclusivo para la autenticación, otro para el catálogo de canciones y álbumes, otro para la subida de archivos y otro para el streaming de audio. Esta separación permitirá que cada parte evolucione de forma independiente, sea más fácil de mantener y tolere mejor errores o cargas altas.

Además, una condición importante del proyecto es la **heterogeneidad tecnológica**. El backend no estará hecho con un solo lenguaje, sino con varios. El servicio de identidad se desarrollará con **Java y Spring Boot**, el servicio de catálogo con **Node.js y Express**, y los servicios de carga y streaming con **Python y FastAPI**. El cliente será una aplicación de escritorio desarrollada con **Electron, React y TypeScript**, por lo que también estará desacoplado del backend. Esta decisión no solo cumple con la restricción académica del proyecto, sino que demuestra cómo pueden convivir distintas tecnologías dentro de una misma solución distribuida.

En cuanto a su funcionamiento, el sistema seguirá un flujo bien definido. Cuando un artista quiera subir una canción, utilizará el cliente de escritorio para seleccionar el archivo de audio y sus imágenes. Antes de enviarlos, la interfaz validará que el tamaño y el tipo sean correctos para prevenir errores desde el inicio. Después, el archivo será enviado al **Upload Service**, que se encargará de almacenar los binarios en **MinIO**, que funcionará como almacenamiento de objetos compartido entre contenedores. Una vez que el archivo quede guardado, este servicio emitirá un evento a través de **RabbitMQ**, informando que el audio fue cargado correctamente. Entonces, el **Catalog Service** escuchará ese evento y actualizará la información del contenido en su propia base de datos sin bloquear el proceso de subida. Esto hace que la escritura pesada esté separada del resto del sistema y sea más tolerante a fallos.

Para la reproducción, el sistema no descargará el archivo completo. En lugar de eso, el **Streaming Service** entregará el audio por partes utilizando **HTTP Range Requests**, devolviendo fragmentos con código **206 Partial Content**. Este punto es central en el proyecto, porque garantiza que el streaming sea real y no una simple descarga disfrazada. Gracias a eso, el usuario podrá empezar a escuchar el contenido sin esperar a que todo el archivo termine de transferirse. También se guardará en **MongoDB** información relacionada con las sesiones y el avance de reproducción, como el offset, lo que permitirá llevar control del estado del streaming.

La comunicación entre componentes también estará dividida según la necesidad. El cliente se comunicará con el sistema por medio de un **API Gateway**, usando peticiones **REST/HTTPS**. Entre microservicios, cuando se necesite velocidad y respuesta inmediata, se usará comunicación síncrona como **gRPC o RPC**, por ejemplo para validar tokens entre el servicio de streaming y el de autenticación. Para tareas que puedan procesarse de forma desacoplada, como la actualización del catálogo después de una carga, se usará comunicación asíncrona mediante **RabbitMQ**. Esto mejora la escalabilidad y reduce la dependencia directa entre servicios.

Desde el punto de vista de la interfaz, la plataforma tendrá una experiencia similar a la de una aplicación de música moderna. Habrá una barra lateral fija para usuarios autenticados y un reproductor inferior permanente, visible en todo momento, para que la reproducción no se interrumpa al cambiar de vista. También se cuidará la prevención de errores desde el cliente, especialmente en formularios relacionados con la carga de archivos. Estas decisiones de diseño se apoyan en las heurísticas de Nielsen, buscando una experiencia más clara, estable y usable.

Finalmente, el proyecto no solo se centrará en funcionar, sino en demostrar calidad técnica. Se buscará que tenga al menos **70% de cobertura de pruebas unitarias**, que pase por análisis estático sin vulnerabilidades críticas y que cumpla métricas de rendimiento, como una latencia de metadatos menor a **300 ms** y un tiempo al primer fragmento de streaming menor a **500 ms**. Además, todo el sistema será **contenedorizado con Docker** y preparado para ejecutarse en **Ubuntu Server**, asegurando que el despliegue sea reproducible y ordenado.

En resumen, **StreamButed será una plataforma distribuida de streaming de audio construida con microservicios, múltiples lenguajes y comunicación híbrida entre servicios, diseñada para ofrecer reproducción eficiente por fragmentos, carga desacoplada de archivos y mayor tolerancia a fallos que un sistema monolítico**. Su propósito será demostrar, en un contexto académico, cómo diseñar e implementar un sistema en red moderno, escalable y alineado con buenas prácticas de arquitectura de software.
